# BX-09-001 - Shipping Fee Calculated Per Seller

## User Story
As a buyer, I want shipping fees calculated per seller so that I know my delivery cost before placing my order.

## Initial Test Before Fixes (Baseline)

Confirm the current behavior before any fix, so the team can clearly see what is missing versus the story.
Scope note: this document is for mobile-app view only, and all checks must include both Android and iOS.

2. Prepare a cart with at least three items from two different sellers.

## Comprehensive Implementation Plan

### Implementation Goal
Move from one checkout-level shipping fee to seller-group shipping fees that are calculated from backend rules and shown clearly in each seller section.

### Why This Approach
1. It matches marketplace behavior where each seller ships separately.
2. It gives buyers clear and fair shipping costs before payment.
3. It reduces wrong totals and order disputes caused by combined shipping.
4. It creates a clean base for BX-09-002 and BX-09-003.

### Mobile Scope and Device Requirements
1. Implementation is for mobile-app view only.
2. UI must stay responsive on common phone widths and heights.
3. Critical actions must be usable on both Android and iOS.
4. No desktop-specific behavior is required in this story plan.

### Reuse and UX Constraints
1. Reuse current checkout modals, sheets, and existing page sections first.
2. Create a new modal or page only when current components cannot support the required seller-level flow clearly and safely.
3. If a new UI surface is truly needed, record the reason and keep the design language consistent with the current mobile app.
4. Prioritize ease of use: fewer taps, clear labels, stable action placement, and immediate feedback for loading and errors.
5. Keep strong validation and calculation accuracy while still making the flow simple and fast for buyers.

### Step-by-Step Plan
1. Audit current checkout UI components and reuse plan first.
Why: this keeps behavior familiar and avoids unnecessary new modal or page creation.

2. Add seller-group shipping state in checkout.
Why: the screen must track fee, loading, and errors per seller, not one global value.

3. Build backend request payload per seller group.
Why: backend needs seller origin, buyer destination, and item package data per group for proper rates.

4. Add shipping calculation service call that returns per-seller results.
Why: ensures system uses backend-calculated values, not only local estimates.

5. Update checkout UI to show seller-level shipping fee blocks.
Why: buyer must see fee under the correct seller section.

6. Add recalculation triggers.
Why: shipping must refresh when buyer changes address, quantity, selected items, or shipping method.

7. Add seller-level failure handling and order blocking.
Why: if one seller shipping is unresolved, checkout must not proceed.

8. Save per-seller shipping fees in checkout payload and order creation flow.
Why: correct values must be used by order service and persisted for later views.

9. Update total calculation to sum per-seller shipping fees.
Why: checkout total must be correct after per-seller calculations complete.

10. Add tests for multi-seller shipping behavior.
Why: prevents regression and confirms rule compliance.

11. Add responsive UI checks for shipping sections.
Why: multi-seller blocks and fee rows must stay readable on small and large phone screens.

12. Add Android and iOS behavior verification.
Why: interaction behavior, loading states, and validation flow must be consistent across both platforms.

### Key Implementation Notes
1. Group by seller id for logic, show seller name for display.
2. Keep loading and error states per seller group.
3. Use clear messages when shipping cannot be calculated.
4. Keep old fallback only as temporary safety net, not final logic.
5. Treat responsiveness and cross-platform usability as release requirements, not optional polish.
6. **Zone detection uses GPS coordinates, not string matching.** The buyer destination zone (NCR, Luzon, Visayas, Mindanao) is resolved from `shipping_addresses.coordinates` (lat/lng already stored by the existing `LocationModal` + `react-native-maps` map-pin flow). The seller origin zone is resolved from `sellers.shipping_origin_lat` / `shipping_origin_lng` (new nullable columns populated when the seller pins their location on the map). This eliminates fragile city-name string comparisons.
7. A pure `getZoneFromCoords(lat, lng)` function using Philippine island-group bounding boxes resolves the zone. When coordinates are absent (seller hasn't pinned yet), a text-based fallback using province/region strings is used instead.

### Risks and How to Reduce Them
1. Risk: wrong total due to mixed old and new logic.
Mitigation: remove old flat shipping branch once new logic is stable.

2. Risk: frequent recalculation can hurt performance.
Mitigation: debounce address and quantity changes before recompute.

3. Risk: partial seller failures create confusing UI.
Mitigation: add seller-level error cards and disable place-order with clear reason.

### Done Criteria
1. Each seller has its own shipping fee shown in checkout.
2. Checkout total equals item subtotal plus all seller shipping fees minus discounts.
3. Invalid address or seller-level shipping failure blocks place order.
4. Recalculation works on all required triggers.
5. Automated and manual tests pass.
6. Shipping section layout remains usable on Android and iOS phone screens.
7. Mobile flow is verified as mobile-only implementation scope.
8. Existing modals and current feature surfaces are reused unless a documented exception is required.
9. Final flow is easy to use while preserving shipping accuracy and validation integrity.
10. **No shipping constants (rates, divisors, thresholds) are hardcoded in any `.ts` file.** All values are fetched from `shipping_config` and `shipping_zones` at runtime.
11. **Zone detection uses GPS coordinates** from react-native-maps, not city/province name strings. The `getZoneFromCoords(lat, lng)` bounding-box function is the primary resolution method.
12. **Full J&T courier math model is applied:** Wv → Wc → Sw → Fv → Total, using all values from the DB.

## Story Reference

## Flow
1. Buyer selects cart items and proceeds to Checkout.
2. System loads the selected delivery address or prompts the buyer to choose one.
3. System identifies the seller group for each selected item.
4. System calculates shipping fee separately for each seller group.
5. System displays the shipping fee under the corresponding seller section.

## Validation
1. System must not calculate one combined shipping fee for all sellers in a multi-seller checkout.
2. System must require a valid delivery address before shipping can be calculated accurately.
3. System must validate that each item is eligible for shipping to the selected destination.
4. System must validate that the seller is active and able to fulfill shipment.

## System Processing
1. System must calculate shipping based on the J&T Express math model: volumetric weight, chargeable weight, weight surcharge, valuation fee (1%), and out-of-delivery-zone fee.
2. **Origin and destination zones** (NCR, Luzon, Visayas, Mindanao) must be resolved from GPS coordinates (`shipping_addresses.coordinates` for buyer, `sellers.shipping_origin_lat/lng` for seller), not from hardcoded city name strings. A bounding-box function resolves zone from lat/lng. Province/region text is a fallback only when coordinates are missing.
3. System must recalculate shipping whenever the buyer changes address, quantity, selected items, or shipping method. Recalculation must be debounced (300ms) to avoid excessive Supabase queries.
4. System must store the shipping fee per seller group in the checkout session.
5. System must use backend-calculated shipping values and must not rely only on client-side values.
6. Rate data is stored in the `shipping_zones` Supabase table (origin zone → destination zone → method). Constants are in the `shipping_config` table. Both tables have public RLS `SELECT` access.

## Display Behavior
1. Shipping fee must be visible under each seller group.
2. System must display a loading state while shipping fees are being calculated.
3. The checkout total must update after shipping calculation is completed.

## Edge Cases
1. If shipping calculation fails for one seller group, system must show an error for that group and prevent order placement until resolved.
2. If no address is selected, system must show a prompt to add or select an address before calculating shipping.
3. If an item cannot be shipped to the selected address, system must clearly identify the affected item or seller group.

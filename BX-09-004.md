# BX-09-004 - Validate Delivery Address Before Order Placement

## User Story
As a buyer, I want my delivery address validated before order placement so that my shipment is not delayed or misrouted.

## Initial Test Before Fixes (Baseline)

### Purpose
Check the current address validation behavior before fixing gaps in required fields, format checks, and serviceability checks.
Scope note: this document is for mobile-app view only, and all checks must include both Android and iOS.

### Test Setup
1. Create test addresses with full data, partial data, and invalid data.
2. Use one buyer account with no saved address and one with multiple saved addresses.
3. Open checkout and address management screens.
4. Run the same baseline test on Android (device or emulator).
5. Run the same baseline test on iOS (device or simulator).

### Baseline Test Cases
1. Required fields validation check.
Expected by story: required fields must be enforced before place order.
Before fix likely result: some required fields are checked, but validation is incomplete across all screens.

2. Phone format validation check.
Expected by story: invalid mobile format should show clear error.
Before fix likely result: weak or inconsistent phone format enforcement.

3. Postal code and address format check.
Expected by story: incomplete or malformed address should be blocked.
Before fix likely result: fallback values like default postal code can allow weak data.

4. Serviceability check.
Expected by story: non-serviceable area should be clearly rejected.
Before fix likely result: serviceability logic may be missing or not consistently enforced.

5. Stale shipping invalidation check.
Expected by story: after editing address to invalid, old shipping values must be cleared.
Before fix likely result: stale shipping values may remain in checkout state.

6. No saved address flow check.
Expected by story: user is prompted to add or select an address before continuing.
Before fix likely result: prompt exists, but validation depth can still be limited.

7. Mobile responsiveness and cross-platform form usability check.
Expected by story: address fields, errors, and action buttons remain readable and usable on Android and iOS phone screens.
Before fix likely result: field spacing, keyboard overlap, or button visibility can differ by platform.

### Baseline Output to Record
1. Error messages shown for each invalid input.
2. Whether place-order is blocked correctly.
3. Whether shipping values are cleared after invalid address edits.
4. Pass, Partial, or Fail per test case.

## Comprehensive Implementation Plan

### Implementation Goal
Implement a strict and clear address validation flow so only valid and serviceable addresses can be used for checkout.

### Why This Approach
1. Good address data is required for accurate shipping fee and ETA.
2. It reduces delivery failures and customer complaints.
3. It prevents stale checkout values after address changes.

### Mobile Scope and Device Requirements
1. Implementation is for mobile-app view only.
2. Address forms and validation messages must be responsive on common phone sizes.
3. Field validation, error rendering, and action flow must be usable on Android and iOS.
4. Desktop and web-specific behavior is out of scope for this file.

### Reuse and UX Constraints
1. Reuse current address form screens, modals, and validation UI patterns first.
2. Create a new modal or page only when existing address flows cannot support required validation clarity.
3. If a new UI surface is required, document the reason and align with the current mobile app behavior.
4. Prioritize ease of use with clear field guidance, quick correction flow, and visible actions.
5. Keep strict validation and serviceability accuracy while minimizing user friction.

### Step-by-Step Plan
1. Audit current address flows and define reuse-first updates before building new UI.
Why: this keeps user behavior familiar and avoids unnecessary new modal or page creation.

2. Create one central checkout address validator in address service.
Why: keeps rules consistent between address form and place-order flow.

3. Define required-field validation rules.
Why: all required fields must be checked the same way everywhere.

4. Add phone and postal format validators.
Why: delivery coordination depends on valid contact and location formats.

5. Add serviceability check call.
Why: valid format alone is not enough if area is not covered.

6. Run validator when saving or editing address.
Why: bad data should be blocked as early as possible.

7. Run validator again before place order.
Why: final safety check prevents invalid or outdated address data.

8. Invalidate and recalculate shipping after address updates.
Why: old shipping values are no longer reliable after address change.

9. Improve error messages in checkout and address screens.
Why: users should know exactly what to fix.

10. Add tests for all invalid and edge scenarios.
Why: ensures behavior stays correct in future updates.

11. Add responsive form checks for small and large mobile screens.
Why: critical actions like save address and place order must remain visible and usable.

12. Add Android and iOS verification passes.
Why: keyboard behavior, field focus, and error display can differ by platform.

### Key Implementation Notes
1. Return structured validation result with valid, errors, and serviceable fields.
2. Keep error messages simple and field-specific.
3. Do not auto-fill weak fallback values in final order payload.
4. Always block place-order if validator returns invalid.
5. Ensure input fields and errors remain clear and actionable on Android and iOS.

### Risks and How to Reduce Them
1. Risk: too strict validation blocks valid addresses.
Mitigation: allow country-specific format variations and test with real samples.

2. Risk: duplicated validation logic between screens.
Mitigation: call one shared validator method from both screens.

3. Risk: stale shipping remains after address edit.
Mitigation: clear shipping state on address change before recalculation.

### Done Criteria
1. Required fields are enforced in address save and place-order.
2. Invalid phone and postal format show clear errors.
3. Non-serviceable addresses are blocked with clear message.
4. Shipping values are invalidated and recalculated after address update.
5. No-saved-address users are guided to add or select an address.
6. Address forms and errors are responsive on Android and iOS phones.
7. Validation and implementation checks are completed under mobile-only scope.
8. Existing address screens and modals are reused unless a documented exception is required.
9. Validation flow is easy to complete while preserving strict data quality and serviceability checks.

### Expected Results After Implementation

#### UI/UX Expected Results
1. Buyers receive clear, field-level guidance using existing address form and checkout surfaces.
2. Validation feedback appears at the right time and in the right place, making corrections easy.
3. Address editing and retry actions are straightforward and do not require confusing navigation.
4. Error messages are simple, specific, and actionable for both Android and iOS users.
5. Form layout remains responsive and usable even with keyboard open on common phone sizes.

#### Functional Expected Results
1. A shared validator enforces required fields, phone format, postal format, and serviceability consistently.
2. Invalid or incomplete addresses are blocked during save and again before place-order.
3. Address updates invalidate stale shipping data and trigger recalculation.
4. Checkout cannot proceed when address validity or serviceability requirements fail.
5. Validation result structure supports reliable behavior across address screens and checkout logic.

## Story Reference

## Flow
1. Buyer selects or adds a delivery address during Checkout.
2. System validates the address details.
3. Buyer reviews the validated shipping information before placing the order.

## Validation
1. Address must include all required fields such as recipient name, mobile number, street or address line, city or municipality, province or state, and postal code where required.
2. System must validate the address format and completeness before order placement.
3. System must validate that the selected address is serviceable based on shipping coverage rules if applicable.
4. System must validate the mobile number format required for delivery coordination.

## System Processing
1. System must store the selected address in the checkout session.
2. System must trigger shipping recalculation after any address update.
3. System must block order placement if the address is incomplete or invalid.

## Display Behavior
1. The selected address must be clearly displayed in the shipping section of Checkout.
2. Validation errors must identify the missing or invalid address fields.
3. Buyer must be able to edit or replace the address before continuing.

## Edge Cases
1. If the address becomes invalid after editing, system must not keep stale shipping values.
2. If the selected area is not serviceable, system must clearly inform the buyer that delivery is unavailable to that address.
3. If buyer has no saved address, system must prompt the buyer to add or select an address before checkout can proceed.

# BX-09-002 - Choose Shipping Method Per Seller

## User Story
As a buyer, I want to choose a shipping method per seller so that I can select my preferred delivery option.

## Initial Test Before Fixes (Baseline)

### Purpose
Check current checkout behavior and confirm gaps before implementing seller-level shipping method selection.
Scope note: this document is for mobile-app view only, and all checks must include both Android and iOS.

### Test Setup
1. Use a buyer account with a valid saved address.
2. Add items from at least two sellers to cart.
3. Open checkout and make sure both seller groups are visible.
4. Run the same baseline test on Android (device or emulator).
5. Run the same baseline test on iOS (device or simulator).

## Comprehensive Implementation Plan

### Implementation Goal
Enable buyer to select a shipping method per seller group, with valid options only, and keep totals and ETA accurate.

### Why This Approach
1. Different sellers and destinations have different couriers and rates.
2. Buyer needs control and transparency per seller shipment.
3. This is required to support accurate ETA and correct final validation.

### Mobile Scope and Device Requirements
1. Implementation is for mobile-app view only.
2. Method picker UI must be responsive on common phone sizes.
3. Tap targets and loading states must be usable on Android and iOS.
4. No desktop or web view behavior is in scope for this file.

### Reuse and UX Constraints
1. Reuse current checkout sections, selectors, and modal patterns before introducing new UI surfaces.
2. Create a new modal or page only when existing components cannot support seller-level method selection clearly.
3. If a new UI surface is required, document the reason and match current mobile app interaction patterns.
4. Prioritize ease of use: simple selection flow, minimal taps, clear selected state, and visible retry path.
5. Keep final behavior both user-friendly and accurate for fee, ETA, and validation outputs.

### Step-by-Step Plan
1. Audit current checkout method UI and define reuse plan first.
Why: keeps buyer flow familiar and avoids unnecessary new modal or page creation.

2. Define data model for selected method per seller group.
Why: system needs a clear map like seller id to selected method.

3. Add service call to fetch available methods per seller group and address.
Why: methods must be dynamic and filtered by real rules.

4. Add UI selector in each seller section.
Why: buyer must choose method directly under the correct seller.

5. Save selection state per seller group.
Why: method must stay selected and visible while user continues checkout.

6. Recalculate seller shipping fee, ETA, and total on selection change.
Why: pricing and delivery details must stay accurate.

7. Add loading and retry behavior for method fetch.
Why: network failures should not trap users without feedback.

8. Add final method revalidation before order placement.
Why: prevents stale or invalid shipping options from being submitted.

9. Block order placement for groups with missing or invalid methods.
Why: ensures checkout integrity and avoids backend rejection.

10. Add tests for selection, revalidation, and failure paths.
Why: protects against regressions after rollout.

11. Add responsive UI checks for method selector blocks.
Why: method name, fee, and ETA rows must remain readable on small phones.

12. Add Android and iOS verification passes.
Why: selection and validation behavior must be consistent across both mobile platforms.

### Key Implementation Notes
1. Keep method data tied to seller id, not display name.
2. Keep fee and ETA attached to selected method object.
3. Show clear state labels: loading, selected, unavailable, retry.
4. Persist selection in checkout session payload.
5. Keep method rows touch-friendly and readable on both Android and iOS.
6. **Method availability is derived from the same coordinate-based zone detection as BX-09-001.** The origin and destination zones (resolved from GPS coordinates via bounding boxes) determine which methods are shown in the picker for each seller group.
7. **Same-day delivery** is only shown when both origin and destination zones are within `same_day_zones` (NCR-only by default, stored in `shipping_config`).
8. **Bulky/Freight** is only shown when the cart's chargeable weight for that seller group meets or exceeds the `bulky_weight_threshold` (default 50 kg, stored in `shipping_config`).
9. The available method list is sourced from the `shipping_zones` Supabase table filtered by `(origin_zone, destination_zone)`. No hardcoded method lists in the mobile app.

### Risks and How to Reduce Them
1. Risk: method list flickers or resets during recalculation.
Mitigation: keep prior selection until new list is confirmed invalid.

2. Risk: wrong seller gets wrong method.
Mitigation: always key state by seller id and verify mapping in tests.

3. Risk: method becomes unavailable between selection and submit.
Mitigation: mandatory revalidation right before place-order API call.

### Done Criteria
1. Buyer can choose shipping method per seller group.
2. Method list is filtered by seller and destination.
3. Totals and ETA update after each selection.
4. Checkout blocks when required methods are missing or invalid.
5. Retry flow works when method fetch fails.
6. Selector and option UI remains usable on Android and iOS screens.
7. All tests are executed under mobile-only scope.
8. Existing modal and selector patterns are reused unless a documented exception is necessary.
9. Method selection flow is easy to use while still enforcing strict checkout validation.
10. **No method availability thresholds or rate constants are hardcoded in app code.** `same_day_zones` and `bulky_weight_threshold` are read from `shipping_config` at runtime.
11. **Available methods per seller are queried from `shipping_zones` table** filtered by GPS-resolved `(origin_zone, destination_zone)` — not from a fixed list in code.

### Expected Results After Implementation

#### UI/UX Expected Results
1. Each seller section shows a clear shipping method selector inside the current checkout experience.
2. Buyers can understand and compare options quickly through readable method name, fee, and ETA display.
3. Selected method state is obvious and remains visible under the correct seller group.
4. Loading and retry behavior is user-friendly and does not interrupt the overall checkout flow.
5. Method selection controls stay tap-friendly and responsive on Android and iOS phone screens.

#### Functional Expected Results
1. System fetches and filters shipping methods per seller group and selected address.
2. Selected method is stored per seller group and remains consistent through checkout updates.
3. Shipping fee, ETA, and total recalculate correctly after method changes using the J&T formula (Wv → Wc → Sw → Fv → Total) with all values from DB.
4. Final validation rechecks method availability before place-order submission.
5. Checkout blocks submission for seller groups with missing, invalid, or stale method selections.
6. **GPS coordinates** (from react-native-maps) determine both which methods are available (zone-pair lookup in `shipping_zones`) and which special methods are eligible (same-day: NCR→NCR, bulky: Wc ≥ threshold from `shipping_config`).

## Story Reference

## Flow
1. Buyer opens Checkout.
2. Buyer views the shipping section under each seller group.
3. Buyer taps the shipping method selector for a seller group.
4. System displays available shipping methods for that seller and destination.
5. Buyer selects one shipping method.
6. System applies the selected method to that seller group.
7. System updates the shipping fee and overall total.

## Validation
1. System must show only the shipping methods available for the selected address and seller group.
2. System must not allow the buyer to proceed without a shipping method if shipping method selection is required.
3. System must validate that the selected shipping option remains available before order placement.

## System Processing
1. System must fetch the available shipping methods dynamically from the `shipping_zones` table, filtered by `(origin_zone, destination_zone)` — where zones are resolved from GPS coordinates as described in BX-09-001.
2. System must filter method list: same-day only for NCR→NCR, bulky only when chargeable weight ≥ threshold; both thresholds come from `shipping_config`.
3. System must save the chosen shipping method per seller group and include it in the checkout payload as `shippingBreakdown[]` with full fee breakdown (base rate, weight surcharge, valuation fee, ODZ fee).
4. System must update shipping fee, ETA, and totals after selection.
5. System must revalidate the selected shipping method during final checkout validation.

## Display Behavior
1. Each shipping option must display courier or service name, shipping fee, and estimated delivery time if available.
2. The selected shipping option must remain visible under the correct seller group.
3. System must show a loading indicator while shipping methods are being fetched.

## Edge Cases
1. If a selected shipping option becomes unavailable before placing the order, system must require the buyer to reselect another valid option.
2. If no shipping method is available for a seller group, system must clearly show that checkout cannot proceed for that group.
3. If shipping methods fail to load, system must allow the buyer to retry.

# Main Task


# BX-09-003 - Show Estimated Delivery Time

## User Story
As a buyer, I want to see the estimated delivery time so that I know when to expect my order.

## Initial Test Before Fixes (Baseline)

### Purpose
Check where ETA is currently shown and where it is missing, before implementing the full story behavior.
Scope note: this document is for mobile-app view only, and all checks must include both Android and iOS.

### Test Setup
1. Use a buyer account with a valid address.
2. Add items from two sellers.
3. Open checkout and proceed until shipping section and order placement are available.
4. Run the same baseline test on Android (device or emulator).
5. Run the same baseline test on iOS (device or simulator).

### Baseline Test Cases
1. Checkout ETA visibility check.
Expected by story: ETA appears near selected shipping option in each seller group.
Before fix likely result: ETA may not appear in checkout seller sections.

2. Multi-seller ETA check.
Expected by story: each seller group shows its own ETA.
Before fix likely result: no per-seller ETA display.

3. ETA refresh on changes check.
Expected by story: ETA refreshes when address or shipping method changes.
Before fix likely result: ETA stays unchanged or is not tied to method updates.

4. Post-order ETA persistence check.
Expected by story: ETA remains visible in order details after order placement.
Before fix likely result: only general delivery date or tracking estimate, not method-linked ETA from checkout.

5. Missing ETA fallback check.
Expected by story: fallback text is shown when ETA is unavailable.
Before fix likely result: blank value, generic date, or inconsistent wording.

6. Mobile responsiveness and cross-platform readability check.
Expected by story: ETA text is readable and well placed in Android and iOS mobile screens.
Before fix likely result: ETA area can be inconsistent in spacing or truncation across platforms.

### Baseline Output to Record
1. Screenshots from checkout, order confirmation, order detail, and tracking.
2. Notes for each case: Pass, Partial, or Fail.
3. Exact locations where ETA is missing or stale.

## Comprehensive Implementation Plan

### Implementation Goal
Show accurate ETA per seller in checkout, refresh it when inputs change, and keep it visible after order placement.

### Why This Approach
1. Buyers decide faster when they know expected delivery time.
2. ETA tied to method and destination reduces confusion.
3. Persistence in order screens improves trust after payment.

### Mobile Scope and Device Requirements
1. Implementation is for mobile-app view only.
2. ETA labels and values must be readable on common phone sizes.
3. Checkout, confirmation, and order detail ETA behavior must be usable on Android and iOS.
4. Desktop-specific behavior is out of scope for this file.

### Reuse and UX Constraints
1. Reuse current checkout and order screen components for ETA display whenever possible.
2. Create a new modal or page only if current components cannot present ETA clearly without confusion.
3. If a new UI surface is needed, document why reuse is not sufficient and keep interaction style consistent.
4. Prioritize ease of use with clear ETA placement, consistent wording, and minimal cognitive load.
5. Keep usability high without reducing ETA accuracy, freshness, or fallback clarity.

### Step-by-Step Plan
1. Audit current ETA display surfaces and define reuse plan first.
Why: this keeps the flow familiar and prevents unnecessary new modal or page creation.

2. Add ETA field into seller shipping option model.
Why: ETA must travel with shipping method data through checkout state.

3. Standardize ETA format function.
Why: one formatting rule avoids mixed texts across screens.

4. Display ETA near each selected method in checkout.
Why: story requires ETA to appear beside method choice.

5. Refresh ETA on address and method changes.
Why: ETA depends on route and service selection.

6. Persist selected ETA in checkout payload and shipment metadata.
Why: order confirmation and detail screens need saved ETA context.

7. Update order mapping layers to read persisted ETA.
Why: UI screens consume mapped order data, not raw payload.

8. Add fallback behavior for missing ETA.
Why: checkout must continue when shipping is valid but ETA is unavailable.

9. Add tests for ETA display, refresh, persistence, and fallback.
Why: verifies complete lifecycle from checkout to post-order views.

10. Add responsive ETA layout checks across mobile screens.
Why: ETA should not overlap, clip, or disappear on smaller devices.

11. Add Android and iOS consistency verification.
Why: ETA display and refresh behavior must match on both mobile platforms.

### Key Implementation Notes
1. Keep ETA value per seller group, not one global ETA.
2. Prefer backend ETA source when available.
3. Use clear fallback text: Delivery estimate unavailable.
4. Do not block checkout only because ETA is missing.
5. Keep ETA text placement stable and readable for Android and iOS users.

### Risks and How to Reduce Them
1. Risk: stale ETA after address update.
Mitigation: invalidate previous ETA and fetch fresh value on change.

2. Risk: different screens show different ETA text.
Mitigation: use one shared formatting helper across checkout and order screens.

3. Risk: ETA missing in order detail due to mapping gap.
Mitigation: include ETA in order mapping tests and snapshots.

### Done Criteria
1. ETA appears near selected shipping option in checkout.
2. ETA is shown per seller group in multi-seller checkout.
3. ETA refreshes after address or method change.
4. ETA or fallback text appears in confirmation and detail screens.
5. Missing ETA does not break checkout flow.
6. ETA UI is responsive on common Android and iOS phone sizes.
7. Verification is completed under mobile-only scope.
8. Current checkout and order UI surfaces are reused unless a documented exception is required.
9. ETA experience remains easy to understand while preserving data correctness.

### Expected Results After Implementation

#### UI/UX Expected Results
1. ETA appears in consistent, easy-to-scan positions in checkout, confirmation, and order detail views.
2. Buyers can quickly understand delivery timing using clear and consistent ETA wording.
3. Fallback text is understandable and does not create uncertainty when ETA is temporarily unavailable.
4. ETA labels and values remain readable on Android and iOS mobile screens without clipping or overlap.
5. The ETA experience stays within current app surfaces and avoids unnecessary new navigation.

#### Functional Expected Results
1. ETA is tied to the selected shipping method and seller group, not a single global value.
2. ETA refreshes correctly when address or shipping method changes.
3. ETA data is persisted from checkout into order-related records for post-order display.
4. Shared formatting logic keeps ETA display consistent across all relevant screens.
5. Missing ETA triggers fallback messaging while allowing checkout to continue when shipping is valid.

## Story Reference

## Flow
1. Buyer selects a shipping method in Checkout.
2. System calculates or retrieves the estimated delivery window.
3. Buyer reviews the estimated delivery time before placing the order.

## Validation
1. ETA must correspond to the selected shipping method and destination.
2. ETA must be shown per seller group if multiple sellers exist.

## System Processing
1. System must retrieve ETA from courier logic, SLA rules, or logistics integration where available.
2. System must refresh ETA when the buyer changes address or shipping method.
3. System must save the ETA associated with the selected shipping option in checkout data.

## Display Behavior
1. ETA must be displayed in readable format such as "2-4 business days" or expected delivery date range.
2. ETA must appear near the selected shipping option.
3. ETA must remain visible in order details after order placement if supported by logistics data.

## Edge Cases
1. If ETA is not available, system must display fallback text such as "Delivery estimate unavailable".
2. Missing ETA must not break checkout if shipping itself is valid.
3. If ETA changes after address or method update, the displayed value must refresh immediately.
